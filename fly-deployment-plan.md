# Fly.io Deployment Plan

## Context

The site currently runs as a fully static Astro 5 build deployed to Netlify. The goal is to migrate to Fly.io to support future SQLite-backed dynamic routes while keeping existing markdown blog posts pre-rendered (fast, no DB cost on every request). The deployment uses Docker + the `@astrojs/node` adapter in standalone mode. Netlify stays live during a testing period before DNS is cut over.

---

## ~~Phase 1 — Switch Astro to hybrid SSR~~ ✅ DONE

**Note:** Astro 5 removed `output: 'hybrid'`. `output: 'static'` now supports per-route SSR opt-in natively — it's the same behaviour. The adapter was added without changing the output mode.

Added `@astrojs/node` adapter (standalone mode) to `astro.config.mjs`. All blog pages remain pre-rendered; future SQLite routes use `export const prerender = false`. Verified locally via `node dist/server/entry.mjs` — all routes returned 200.

## Phase 2 — Docker files

### 2a. Create `Dockerfile` (multi-stage)

Uses Node 24 (matches `.nvmrc`). Builder stage installs deps and builds; runtime stage is lean.

```dockerfile
FROM node:24-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:24-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./package.json
ENV HOST=0.0.0.0
ENV PORT=8080
EXPOSE 8080
CMD ["node", "./dist/server/entry.mjs"]
```

### 2b. Create `.dockerignore`

Keeps secrets and build artifacts out of the image:

```
node_modules
dist
.env
.env.*
.DS_Store
.astro
```

## Phase 3 — Fly.io app setup

### 3a. Create the Fly app (no deploy yet)
```bash
fly launch --no-deploy
```
This creates the app, assigns a name, and generates a base `fly.toml`. When prompted, decline auto-generated Dockerfile (we have our own).

### 3b. Edit `fly.toml`

Replace generated content with:

```toml
app = '<your-app-name>'        # set by fly launch
primary_region = 'iad'         # US East; change if preferred

[build]

[env]
  PORT = "8080"

[http_service]
  internal_port = 8080
  force_https = true
  auto_stop_machines = "stop"   # machine stops when idle (saves ~$1.94/mo)
  auto_start_machines = true    # wakes on incoming request
  min_machines_running = 0

[[vm]]
  memory = "256mb"
  cpu_kind = "shared"
  cpus = 1

[mounts]
  source = "sqlite_data"
  destination = "/data"         # mount point for future SQLite DB file
```

### 3c. Create the persistent volume (for future SQLite)
```bash
fly volumes create sqlite_data --region iad --size 1
```
1 GB is plenty for a SQLite blog DB. Costs ~$0.15/mo.

### 3d. Handle environment variables
Any values from `.env` that the app needs at runtime should be set as Fly secrets rather than baked into the image:
```bash
fly secrets set KEY=value
```

## Phase 4 — Deploy and verify on fly.dev

```bash
fly deploy
```

Verify:
- `https://<appname>.fly.dev` loads the home page
- A blog post (e.g. `/posts/git-alias/`) renders correctly with code highlighting
- `/posts/` index lists all posts
- `/about/` loads

---

## Phase 5 — DNS cutover (when ready)

After testing the fly.dev URL for as long as desired:

### 5a. Add the domain to Fly
```bash
fly certs add shealeslein.com
fly certs add www.shealeslein.com
```

### 5b. Get the IP addresses
```bash
fly ips list
```

### 5c. Update DNS at your registrar
- Point `A` record for `shealeslein.com` to the IPv4 address from Fly
- Point `AAAA` record to the IPv6 address
- For `www`: `CNAME www → <appname>.fly.dev` (or same A/AAAA)

### 5d. Monitor cert provisioning
```bash
fly certs check shealeslein.com
```
TLS certs auto-provision via Let's Encrypt once DNS propagates.

### 5e. Decommission Netlify
Once `shealeslein.com` resolves to Fly and HTTPS works, disable or delete the Netlify site.
